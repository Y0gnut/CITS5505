Reflective Report: AI-Assisted Development Process

During this project I used GitHub Copilot Chat and ChatGPT as development assistants, not as automatic code replacement tools. I mainly used them for three tasks: drafting first-pass JavaScript logic, diagnosing edge-case bugs, and reviewing alternative implementation strategies. For example, I asked for suggestions on AJAX data loading, dynamic rendering, and event handling patterns. I then compared outputs with assignment requirements, browser behavior, and MDN documentation before integrating anything. This workflow helped me move faster, but still required technical judgment to verify correctness, improve code quality, and keep the final implementation aligned with academic and engineering standards.

Use Case 1: Code Generation (Quiz AJAX Logic)

Example Prompt Sent to AI

Create jQuery quiz logic that loads questions from data/questions.json using $.ajax(),
shuffles with Fisher-Yates, renders each question with 4 radio options, validates all answers on submit,
and shows score + pass/fail without page reload.

What the AI Produced

The AI returned a workable structure with an AJAX request, a shuffle function, and score calculation logic. It also proposed rendering question blocks in a loop and collecting answers on submit. This was a useful starting point because it outlined the main control flow quickly and reduced blank-page time.

What I Changed and Why

I replaced string-concatenated HTML rendering with explicit createElement/appendChild construction to satisfy assignment requirements and reduce markup mistakes. I also added stricter JSON shape validation because the original suggestion assumed all records were valid. Finally, I wrapped localStorage history operations in try/catch for resilience in restricted browsing modes. These changes were made to improve robustness, compliance, and maintainability rather than just making the code "work" once.

Use Case 2: Debugging (beforeunload Not Clearing)

Example Prompt Sent to AI

My quiz attaches beforeunload after first answer, but warning still appears after successful submit.
Show how to remove the listener correctly and avoid duplicate attachments.

What the AI Suggested

The AI suggested storing the event handler as a named function, tracking whether the listener had already been added, and explicitly calling removeEventListener after successful submission. It also suggested separating attachment and detachment into dedicated helper functions.

What I Changed

I implemented attachBeforeUnload() and detachBeforeUnload() with a boolean flag (beforeUnloadAttached) so the listener is never registered twice. On valid submission I call detach before showing results and saving attempt history. I then re-tested by answering one question, submitting successfully, and refreshing/navigating away to confirm no warning appears.

Critical Evaluation

Subtle AI Error I Found

In one iteration, the AI suggested using event.returnValue = true as the key mechanism for the beforeunload warning and implied custom text would be shown. This is subtly wrong for modern browsers: custom strings are ignored, and the reliable pattern is calling event.preventDefault() plus setting event.returnValue to an empty string for compatibility.

How I Detected the Error

I tested in Chrome and noticed the suggested custom message never appeared. I then checked MDN documentation for beforeunload, which confirms browser limitations and recommended usage. The mismatch between AI output and real browser behavior exposed that the suggestion mixed older patterns with modern behavior assumptions.

Broader Limitations and Risks

This experience highlighted common AI limitations: hallucination risk (invented certainty), outdated API knowledge, and over-confident tone even when details are version-sensitive. AI can accelerate ideation and boilerplate generation, but it cannot be treated as an authority source. Every non-trivial suggestion must be tested and cross-checked against trusted references.

Ethical Reflection

I treated AI output as a draft assistant and kept responsibility for design, verification, and final decisions. This supports academic integrity because understanding and accountability remain with me. Over-reliance would risk submitting code I cannot explain, which is both an ethical and practical problem in software engineering.

Conclusion

I learned that AI is most valuable as a rapid collaborator for drafting, idea comparison, and debugging hypotheses, not as a substitute for engineering judgment. The biggest gains came from iterative prompting plus strict verification through browser testing and documentation checks. I also became more careful about edge cases, especially event lifecycles and API response validation. Going forward, I will keep using AI to accelerate routine work, while preserving manual review, testing discipline, and ownership of final technical decisions.
