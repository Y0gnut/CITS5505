HTML: Semantic Structure and Why It Matters

Semantic HTML gives meaning to page regions. A screen reader can quickly jump to a nav, understand where the main content begins, and identify complementary content in an aside. Search engines also interpret this structure to understand your content hierarchy, which improves discoverability.

Good document structure reduces technical debt: teammates can scan markup faster, CSS selectors stay more predictable, and JavaScript hooks become explicit. The result is fewer fragile workarounds and more maintainable code over time.

Annotated Example: Semantic Page Skeleton

<!doctype html>
<html lang="en">
  <head>
    <meta charset="UTF-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1.0" />
    <title>Student Portfolio</title>
  </head>
  <body>
    <header>
      <h1>Alex Student</h1>
      <nav aria-label="Primary">
        <ul>
          <li><a href="#projects">Projects</a></li>
          <li><a href="#contact">Contact</a></li>
        </ul>
      </nav>
    </header>

    <main id="content">
      <article aria-labelledby="post-title">
        <h2 id="post-title">Building an Accessible Quiz</h2>
        <p>Semantic landmarks help keyboard and screen-reader users navigate quickly.</p>
      </article>

      <aside>
        <h3>Related Resources</h3>
        <ul>
          <li><a href="https://developer.mozilla.org">MDN Docs</a></li>
        </ul>
      </aside>
    </main>

    <footer>
      <p>Copyright 2026 Alex Student</p>
    </footer>
  </body>
</html>

CSS: Selectors, Layout, and Responsive Thinking

CSS selectors are not only for styling; they encode intent. A selector like .card-list > .card communicates structure, while utility classes can communicate behavior. Choose selectors that balance clarity and flexibility so your styles are easy to extend.

The box model defines spacing rules through content, padding, border, and margin. Many layout bugs come from misunderstanding where spacing belongs. Consistent rules reduce unpredictable overflow and visual misalignment.

Flexbox and Grid solve different problems: Flexbox aligns content along one axis; Grid manages two-dimensional tracks. For responsive design, start mobile-first, use relative units, and add breakpoints only where layout truly breaks. This leads to simpler CSS and better long-term maintainability.

Annotated Example: Responsive Card Grid

/* Base reset that prevents width overflow from padding/border */
* {
  box-sizing: border-box;
}

/* Semantic parent: each child card gets the same layout rules */
.card-list {
  display: grid;
  grid-template-columns: 1fr;
  gap: 1rem;
  margin: 0;
  padding: 0;
  list-style: none;
}

.card {
  padding: 1rem;
  border: 1px solid #2b4a75;
  border-radius: 0.75rem;
  background: linear-gradient(145deg, #11233d, #0a1a30);
}

/* At 768px+, switch to two columns */
@media (min-width: 48rem) {
  .card-list {
    grid-template-columns: repeat(2, minmax(0, 1fr));
  }
}

/* At 1200px+, use three columns and slightly larger spacing */
@media (min-width: 75rem) {
  .card-list {
    grid-template-columns: repeat(3, minmax(0, 1fr));
    gap: 1.25rem;
  }
}

JavaScript: DOM, Events, and Async Design

DOM manipulation should be intentional and minimal. Querying elements repeatedly inside loops or re-rendering large sections can cause performance issues. Cache references, update only what changed, and separate rendering from business logic so behavior remains predictable.

Events are the backbone of browser interactivity. Delegation matters because it lets one listener handle many dynamic elements, reducing memory usage and keeping code DRY. Good event architecture also prevents bugs caused by duplicated listeners.

Asynchronous JavaScript with fetch and async/await keeps interfaces responsive while waiting for network or storage operations. Proper error handling matters because silent failures make debugging difficult and create confusing user experiences.

Annotated Example: Event Delegation + Async Data Fetch

// Cache parent container once instead of attaching many listeners.
const list = document.querySelector("#task-list");
const status = document.querySelector("#status");

// One delegated event handler works for current and future buttons.
list.addEventListener("click", (event) => {
  if (!event.target.matches("button[data-remove]")) {
    return;
  }

  const row = event.target.closest("li");
  if (row) {
    row.remove();
  }
});

async function loadTasks() {
  status.textContent = "Loading tasks...";

  try {
    const response = await fetch("/api/tasks");
    if (!response.ok) {
      throw new Error(`HTTP ${response.status}`);
    }

    const tasks = await response.json();
    list.innerHTML = tasks
      .map(
        (task) =>
          `<li>${task.title} <button data-remove>Remove</button></li>`
      )
      .join("");

    status.textContent = `Loaded ${tasks.length} tasks.`;
  } catch (error) {
    status.textContent = `Unable to load tasks: ${error.message}`;
  }
}

loadTasks();
